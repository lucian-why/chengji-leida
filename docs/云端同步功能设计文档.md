## 一、需求背景

成绩雷达目前数据存储在浏览器/小程序的 **localStorage** 中，用户换设备或清缓存后数据会丢失。需要实现**云端同步**功能，让用户可以在多个设备间安全地备份和恢复成绩数据。

### 核心原则

- **手动同步**：不追求实时，通过用户主动操作触发
- **以档案为粒度**：档案（Profile）是数据操作的最小单元
- **用户完全掌控**：用户可查看云端数据状态，自主决定备份/恢复/删除


**档案是天然的隔离和同步单元。**

## 三、技术方案

### 3.1 云端存储方案：Supabase

使用 Supabase 作为后端，原因：
- 免费额度足够个人项目使用
- 内置用户认证（Auth）
- 提供 PostgreSQL 数据库 + REST API
- 支持 RLS（行级安全策略）
- 已有登录认证基础（`auth.js`）

### 3.2 数据库表设计

#### 表：`cloud_profiles`（云端档案）

```sql
CREATE TABLE cloud_profiles (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  profile_id VARCHAR(50) NOT NULL,           -- 本地档案 ID
  profile_name VARCHAR(100) NOT NULL,         -- 档案名称（方便预览）
  profile_data JSONB NOT NULL,                -- 完整档案数据（含设置等）
  exam_count INTEGER DEFAULT 0,               -- 该档案下考试数量（冗余，用于预览）
  data_size INTEGER DEFAULT 0,                -- 数据大小字节（用于预览）
  last_sync_at TIMESTAMPTZ DEFAULT now(),     -- 最后同步时间
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

-- 索引
CREATE INDEX idx_cloud_profiles_user ON cloud_profiles(user_id);
CREATE INDEX idx_cloud_profiles_user_pid ON cloud_profiles(user_id, profile_id);

-- RLS：用户只能访问自己的数据
ALTER TABLE cloud_profiles ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can manage own profiles"
  ON cloud_profiles FOR ALL
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);
```

**为什么用一个表而不是拆成档案表+考试表？**

因为：
1. 单个用户的数据量极小（几十 KB），不需要规范化
2. 备份/恢复时一次读写即可，减少 API 调用次数
3. 以档案为粒度操作，JSONB 天然适合存储嵌套数据
4. 避免跨表事务的复杂度

#### 表：`cloud_sync_log`（同步日志，可选）

```sql
CREATE TABLE cloud_sync_log (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  action VARCHAR(20) NOT NULL,               -- 'upload' | 'download' | 'delete'
  profile_ids TEXT[],                        -- 操作涉及的档案 ID 列表
  details JSONB,                             -- 操作详情
  created_at TIMESTAMPTZ DEFAULT now()
);

ALTER TABLE cloud_sync_log ENABLE ROW LEVEL SECURITY;
```

日志表用于调试和问题排查，非必须，可后期添加。

## 四、同步逻辑详解

### 4.1 合并策略：智能合并

当「从云端恢复」时，本地和云端同一档案下都有考试数据时的处理：

```
对于每个考试（按 examId 匹配）：
├── 只在本地有      → 保留（不受影响）
├── 只在云端有      → 合并到本地
└── 两边都有        → 比较 updated_at 时间戳
                       ├── 云端更新 → 用云端的覆盖本地的
                       └── 本地更新 → 保留本地的（不覆盖）
```

**伪代码：**

```javascript
function mergeExams(localExams, cloudExams) {
  const examMap = new Map();
  
  // 先加入所有本地考试
  for (const exam of localExams) {
    examMap.set(exam.id, { source: 'local', data: exam });
  }
  
  // 用云端考试合并（同 ID 则比较时间戳）
  for (const exam of cloudExams) {
    const existing = examMap.get(exam.id);
    if (!existing) {
      // 云端独有的 → 直接加入
      examMap.set(exam.id, { source: 'cloud', data: exam });
    } else if (new Date(exam.updated_at) > new Date(existing.data.updated_at)) {
      // 云端的更新 → 用云端的
      examMap.set(exam.id, { source: 'cloud', data: exam });
    }
    // 否则保留本地更新的
  }
  
  // 按考试开始时间排序返回
  return Array.from(examMap.values())
    .map(item => item.data)
    .sort((a, b) => new Date(a.date) - new Date(b.date));
}
```

### 4.2 三大操作

#### 📤 备份到云端（Upload）

```
用户勾选本地档案 → 点击「备份选中」
  ↓
遍历选中的档案：
  ├── 收集该档案的所有考试数据
  ├── 序列化为 JSONB
  └── UPSERT 到 cloud_profiles 表
       （按 user_id + profile_id 唯一确定一条记录）
  ↓
显示结果：✅ 成功备份 N 个档案
```

**特点：全量覆盖模式**
- 本地有什么就传什么，云端对应档案被完整替换
- 本地删除了某门考试 → 备份后云端也没了
- 不需要"已删除"标记机制

#### 📥 从云端恢复（Download / Merge）

```
用户勾选云端档案 → 点击「恢复选中」
  ↓
遍历选中的云端档案：
  ├── 从 cloud_profiles 取出该档案数据
  ├── 与本地同 ID 档案做智能合并（见 4.1）
  └── 写入 localStorage
  ↓
刷新页面数据显示
  ↓
显示结果：✅ 成功恢复 N 个档案（新增 X 门，更新 Y 门）
```

**特点：智能合并不覆盖**
- 云端独有的考试 → 新增到本地
- 两边都有的 → 取最新的
- 本地独有的 → 不受影响

#### 🗑️ 删除云端数据（Delete）

```
用户勾选云端档案 → 点击「删除云端选中」
  ↓
二次确认：「确定要删除选中的 N 个档案吗？此操作不可撤销！」
  ↓
从 cloud_profiles 物理删除对应记录
  ↓
本地数据不受影响
  ↓
显示结果：✅ 已删除 N 个云端的档案
```

## 五、UI 设计

### 5.1 入口位置 ✅ 已确认：设置页 - 数据管理卡片

**决策**：放在 **设置页 → 数据管理卡片** 内，在「导出数据」和「导入数据」下方新增「☁️ 云端同步」入口项。

**交互方式**：点击后弹出同步面板弹窗（Modal/Popup），不影响设置页原有布局。

```
设置页
├── 📁 档案管理（Card）
│   ├── 当前档案选择器
│   └── 新建 / 重命名 / 删除档案
│
└── 💾 数据管理（Card）          ← 在这个卡片里
    ├── ➕ 添加示例数据
    ├── 📤 导出数据（Excel）    ← 原有
    ├── 📥 导入数据（Excel）    ← 原有
    └── ☁️ 云端同步            ← 新增入口，点击打开同步面板
```

**选择理由**：
- 与导入导出功能天然归类在一起，用户心理模型一致（都是"数据进出"操作）
- 不占用侧栏宝贵空间，不增加主界面复杂度
- 设置页本身是低频操作页面，放这里不会干扰日常成绩录入流程
- 弹窗模式让同步操作有明确的"开始/结束"边界感，用户不会迷失

### 5.3 状态指示

|------|------|
| ✅ 双方一致 | 本地和云端该档案数据量相同、时间接近 |

| ☁️ 仅云端 | 云端有但本地没有（可能在新设备上） |
| 📂 仅本地 | 本地有但还没备份过 |

### 5.4 交互细节

1. **未登录状态**：显示「请先登录」按钮，点击调用已有登录流程
2. **加载中**：操作期间按钮 disabled + loading 动画 + 文字提示
3. **成功提示**：Toast 提示操作结果（如"成功备份 2 个档案"）
4. **错误提示**：Toast 显示具体错误信息（如"网络连接失败"）
5. **删除二次确认**：弹出 modal 确认框，列出要删除的档案名称
6. **全选/取消全选**：每侧都有全选 checkbox

## 八、错误处理与边界情况

| 场景 | 处理方式 |
|------|---------|
| 网络断开/超时 | catch 错误 → Toast 提示"网络异常，请检查网络" |
| Token 过期 | Supabase client 自动 refresh token，失败则提示重新登录 |
| 云端数据被篡改/损坏 | 校验 JSON 结构，无法解析则提示"云端数据异常"并跳过该档案 |
| 单个档案超过限制（如 > 5MB） | 提示"该档案数据过大"，建议清理后再试 |
| 并发操作（快速连点） | 按钮 loading 状态防重复提交 |
| 用户从未备份过 | 右侧显示空状态："还没有备份数据，快去备份吧~" |
| 小程序环境兼容 | 使用 wx.request 封装而非直接 fetch，或使用 Supabase JS SDK 的适配层 |

## 九、小程序适配注意事项

小程序版（`成绩雷达_小程序/`）同样需要此功能，需额外注意：

1. **请求域名白名单**：需在微信公众平台添加 Supabase 域名
2. **echarts-for-weixin 冲突**：避免引入过大的包影响体积
3. **Storage 限制**：小程序 localStorage 有 10MB 限制，大档案需要注意
4. **登录态管理**：小程序使用 wx.login + 自定义登录，与 Web 的 Supabase Auth 不同，需要统一或分别处理

建议：**先完成 Web 版同步功能，验证流程后再适配小程序。**

## 附录：关键决策记录

| 决策点 | 结论 | 理由 |
|--------|------|------|
| 实时 vs 手动 | ✅ 手动同步 | 个人工具场景，不需要实时；手动实现简单可靠 |
| 全量 vs 增量 | ✅ 全量（单档案维度） | 数据量极小（几十KB），没必要增量同步 |
| 合并 vs 覆盖下载 | ✅ 智能合并 | 保护本地数据不被意外覆盖 |
| 上传模式 | ✅ 全量覆盖 | 简化删除问题，本地删了什么云端就同步删除 |
| 操作粒度 | ✅ 以档案为单位 | 档案是天然的数据分组单元，比逐条选考试合理 |
| 存储方案 | ✅ Supabase JSONB | 已有账号体系；单用户数据小无需拆表；减少API调用 |
| 是否需要逐条选择 | ❌ 不需要 | 数据量小无意义；考试间有依赖关系不宜拆开 |
| UI 入口位置 | ✅ 设置页 → 数据管理卡片 | 与导入导出归类一致，点击弹窗打开同步面板（2026-04-04 确认） |
